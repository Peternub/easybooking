ТЕХНИЧЕСКОЕ ЗАДАНИЕ (ТЗ) - ФИНАЛЬНАЯ ВЕРСИЯ
Наименование проекта: Автоматизированная система бронирования услуг (Telegram Mini App)
Стек: Supabase (БД) + Google Calendar (визуализация) + Telegram Bot API + React Mini App
Пакетный менеджер: bun
Линтер: Biome
Хостинг: Vercel
Часовой пояс: Москва (Europe/Moscow)

═══════════════════════════════════════════════════════════════════════════════

1. ОБЩЕЕ ОПИСАНИЕ СИСТЕМЫ
Сервис для онлайн-записи клиентов салона красоты через Telegram Mini App.

Архитектура:
- Supabase: Основное хранилище данных (мастера, услуги, записи, отзывы)
- Google Calendar: Визуализация расписания и проверка занятости (у каждого мастера свой календарь)
- Telegram Bot: Уведомления и управление
- React Mini App: Интерфейс бронирования

2. АРХИТЕКТУРА ДАННЫХ

2.1. SUPABASE - СТРУКТУРА БД

Таблица: masters
- id (uuid, PK)
- name (text) - ФИО
- photo_url (text) - URL фото
- description (text) - Описание
- specialization (text) - Специализация
- google_calendar_id (text) - ID календаря Google
- is_active (boolean) - Активен ли мастер
- created_at (timestamp)

Таблица: services
- id (uuid, PK)
- name (text) - Название услуги
- description (text) - Описание
- price (integer) - Цена в рублях
- duration_minutes (integer) - Длительность в минутах
- category (text) - Категория
- is_active (boolean)
- created_at (timestamp)

Таблица: master_services (связь многие-ко-многим)
- master_id (uuid, FK -> masters)
- service_id (uuid, FK -> services)
- PRIMARY KEY (master_id, service_id)

Таблица: master_schedules (график работы)
- id (uuid, PK)
- master_id (uuid, FK -> masters)
- day_of_week (integer) - 0=Пн, 6=Вс
- start_time (time) - Начало рабочего дня
- end_time (time) - Конец рабочего дня
- is_working (boolean) - Работает ли в этот день

Таблица: master_schedule_exceptions (отпуска, выходные)
- id (uuid, PK)
- master_id (uuid, FK -> masters)
- date (date) - Дата исключения
- reason (text) - Причина (отпуск, больничный и т.д.)
- created_at (timestamp)

Таблица: bookings (записи)
- id (uuid, PK)
- client_telegram_id (bigint) - Telegram ID клиента
- client_name (text) - Имя клиента
- client_username (text) - Username клиента
- master_id (uuid, FK -> masters)
- service_id (uuid, FK -> services)
- booking_date (date) - Дата записи
- booking_time (time) - Время записи
- status (text) - 'active', 'completed', 'cancelled_by_client', 'cancelled_by_admin'
- cancellation_reason (text) - Причина отмены (если отменил админ)
- google_event_id (text) - ID события в Google Calendar
- created_at (timestamp)
- updated_at (timestamp)

Таблица: reviews (отзывы)
- id (uuid, PK)
- booking_id (uuid, FK -> bookings)
- client_telegram_id (bigint)
- master_id (uuid, FK -> masters)
- service_id (uuid, FK -> services)
- rating (integer) - Оценка от 1 до 5
- comment (text) - Текстовый отзыв
- created_at (timestamp)

Таблица: admins (администраторы)
- telegram_id (bigint, PK)
- name (text)
- added_at (timestamp)

2.2. GOOGLE CALENDAR
- У каждого мастера отдельный календарь
- События содержат: дату, время, имя клиента, услугу
- Используется для проверки занятости и визуализации
- Администратор может вручную создавать события (обед, перерыв) - они блокируют время

3. ПОЛЬЗОВАТЕЛЬСКИЕ СЦЕНАРИИ: КЛИЕНТ

3.1. ПРОЦЕСС БРОНИРОВАНИЯ (Mini App)

Шаг 1: Выбор услуги
- Отображается список активных услуг из Supabase
- Показывается: название, описание, цена, длительность

Шаг 2: Выбор мастера
- Показываются только мастера, которые оказывают выбранную услугу
- Отображается: фото, имя, специализация

Шаг 3: Выбор даты и времени
- Система загружает график работы мастера из Supabase
- Проверяет исключения (отпуска) в master_schedule_exceptions
- Проверяет занятость через Google Calendar API
- Показывает только свободные слоты
- ОГРАНИЧЕНИЕ: Нельзя записаться менее чем за 1 час до начала услуги

Шаг 4: Подтверждение
- Итоговый экран с деталями: услуга, мастер, дата, время, цена
- Кнопка "Записаться"

3.2. СОЗДАНИЕ ЗАПИСИ

При нажатии "Записаться":
1. Проверка на двойное бронирование (транзакция в Supabase)
2. Если слот занят → ошибка обоим клиентам
3. Если свободен:
   - Создается запись в Supabase (статус 'active')
   - Создается событие в Google Calendar мастера
   - Клиент получает уведомление в Telegram с деталями

3.3. ОТМЕНА ЗАПИСИ КЛИЕНТОМ

- В уведомлениях (за 24ч и за 1ч) есть кнопка "Отменить запись"
- При нажатии:
  - Статус в Supabase меняется на 'cancelled_by_client'
  - Событие удаляется из Google Calendar
  - Администратор получает уведомление об отмене

3.4. СИСТЕМА ОТЗЫВОВ

- Через 2 часа после начала услуги клиент получает уведомление в бот
- Сообщение содержит кнопку для перехода в Mini App
- В Mini App открывается форма отзыва:
  - Оценка от 1 до 5 звезд
  - Текстовое сообщение (необязательно)
- Отзыв сохраняется в таблицу reviews

4. ПОЛЬЗОВАТЕЛЬСКИЕ СЦЕНАРИИ: АДМИНИСТРАТОР

4.1. ИДЕНТИФИКАЦИЯ
- Администратор определяется по Telegram ID из таблицы admins
- Изначально один администратор (ID указывается при настройке)

4.2. УПРАВЛЕНИЕ ЧЕРЕЗ MINI APP
Администратор видит дополнительные разделы:
- Управление услугами (добавить/редактировать/деактивировать)
- Управление мастерами (добавить/редактировать/деактивировать)
- Управление графиком работы
- Просмотр всех записей
- Отмена записей с указанием причины

4.3. ОТМЕНА ЗАПИСИ АДМИНИСТРАТОРОМ
При отмене:
- Статус в Supabase меняется на 'cancelled_by_admin'
- Сохраняется причина отмены в cancellation_reason
- Событие удаляется из Google Calendar
- Клиент получает уведомление с причиной отмены

4.4. УПРАВЛЕНИЕ ВЫХОДНЫМИ
Два способа:
1. Плановые (отпуска): Добавляются в master_schedule_exceptions через админ-панель
2. Экстренные (больничный): Создается событие в Google Calendar вручную

4.5. УВЕДОМЛЕНИЯ ДЛЯ АДМИНИСТРАТОРА
Администратор получает уведомления о:
- Новых записях
- Отменах клиентами
- Новых отзывах

═══════════════════════════════════════════════════════════════════════════════

5. СИСТЕМА АВТОМАТИЧЕСКИХ УВЕДОМЛЕНИЙ

Бот проверяет базу данных по расписанию (cron jobs):

5.1. Напоминание за 24 часа
- Сообщение с деталями: услуга, мастер, дата, время, адрес салона
- Кнопка "Отменить запись"

5.2. Напоминание за 1 час
- Короткое напоминание
- Кнопка "Отменить запись"

5.3. Запрос отзыва (через 2 часа после начала)
- Сообщение с благодарностью
- Кнопка для перехода в Mini App для оставления отзыва

6. ТЕХНИЧЕСКИЕ ТРЕБОВАНИЯ

6.1. ТЕХНОЛОГИЧЕСКИЙ СТЕК
- Backend: Node.js + Telegram Bot API
- Frontend: React + Vite + TypeScript
- UI: @telegram-apps/telegram-ui (официальная библиотека Telegram)
- БД: Supabase (PostgreSQL)
- Календарь: Google Calendar API
- Пакетный менеджер: bun
- Линтер/Форматтер: Biome
- Хостинг: Vercel

6.2. БЕЗОПАСНОСТЬ
- Авторизация через Telegram WebApp initData
- Проверка подписи данных от Telegram
- Row Level Security (RLS) в Supabase
- Разграничение прав: клиент / администратор

6.3. ОБРАБОТКА КОНФЛИКТОВ
- При двойном бронировании: транзакция с проверкой уникальности
- Если слот занят → оба клиента получают ошибку "Слот уже занят"

6.4. ОТКАЗОУСТОЙЧИВОСТЬ
- При недоступности Google Calendar API:
  - Показывать понятное сообщение пользователю
  - Использовать данные из Supabase как fallback
  - Логировать ошибки для администратора

6.5. СИНХРОНИЗАЦИЯ
- При создании записи: сначала Supabase, потом Google Calendar
- При отмене: обновление в обеих системах
- Supabase - источник истины (Master Data)
